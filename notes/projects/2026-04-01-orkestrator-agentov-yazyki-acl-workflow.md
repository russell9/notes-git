# @context

**Постановка:** агент-оркестратор собирает структурированные результаты от нескольких агентов, агрегирует и проверяет их, затем либо завершает выдачей ответа, либо **рекуррентно** инициирует новые запросы к подагентам с уточнённым контекстом. Нужно понять, **как формализовать** такое поведение и **какие языки/нотации** для этого существуют.

**Исходный черновик:** в «Основное» уже набросана практическая модель (supervisor, граф состояний, BPMN, ACL, JSON Schema и т.д.); ниже добавлен обзор литературы и смежных направлений.

## @goal

Систематизировать описание **оркестратора над несколькими агентами**: научные и инженерные рамки для **диалога между агентами**, **циклов «критика → повтор»**, **workflow**; указать смежные области (верификация терминации, лимиты итераций, формальные спецификации).

## Как мыслить об агенте-оркестраторе

Такого агента удобно описывать не как «один умный LLM», а как **контроллер над графом состояний**:

1. **Входы** — результаты от агентов A, B, C (структурированные: JSON/схема).
2. **Состояние** — что уже известно, что не хватает, лимиты (итерации, время, бюджет токенов).
3. **Правило остановки** — когда считать ответ готовым (все поля заполнены, проверки прошли, confidence выше порога).
4. **Правило повтора** — при каких условиях **переформулировать запрос** к какому агенту и с каким контекстом (часто это «replan» или «critique → revise»).
5. **Инварианты** — нельзя бесконечно крутиться: максимум N циклов, backoff, эскалация человеку.

В литературе и проде это близко к **supervisor/orchestrator**, **reflective / self-correcting loop**, **multi-agent debate → judge**, **map-reduce over agents**.

## Есть ли «языки описания действий агентов»

Да, но это разные уровни: **обмен сообщениями**, **оркестрация workflow**, **формальная логика**, **код**.

### 1) Сообщения между агентами (ACL / протоколы)

- **FIPA ACL** — классический «agent communication language» (performative + ontology), больше академический/стандартный контекст.  
  Документация: [FIPA ACL Message Structure Specification](https://fipa.org/specs/fipa00061/SC00061G.html)
- Исторически рядом **KQML** — ранний язык сообщений для агентов (сейчас реже в проде).

На практике чаще **не отдельный ACL**, а **JSON-схема** + роли + контракт API.

### 2) Языки/модели оркестрации (граф, шаги, ветвления, циклы)

- **BPMN 2.0** — визуально и формально описывает процесс с ветвлениями, циклами, компенсациями (хорошо для «бизнес-процесс + люди/сервисы»).  
  Обзор: [BPMN 2.0 (OMG)](https://www.omg.org/spec/BPMN/2.0/)
- **Temporal / Cadence workflows** (код на Go/Java/TS/Python) — долгоживущие процессы, ретраи, таймеры, человек в контуре.  
  Сайт: [Temporal](https://temporal.io/)
- **Облачные state machines**: AWS Step Functions, Azure Logic Apps, Google Workflows — YAML/JSON + шаги, ветвления, ретраи.
- **В экосистеме LLM** часто **граф как код**: LangGraph, CrewAI flows, AutoGen patterns — это не один «стандартный язык», но **явный граф состояний** (узлы = агенты/инструменты, рёбра = условия).

### 3) Формальные спецификации поведения

- **Statecharts** (Harel) / **UML state machines** — явные состояния и переходы по событиям (удобно для «ждём ответы → агрегируем → решаем повторить»).
- **TLA+ / PlusCal** — если нужна строгая проверка свойств (нет зависаний, корректность терминации).  
  Сайт: [TLA+](https://lamport.azurewebsites.net/tla/tla.html)
- **Petri nets** — когда важны параллельность и ресурсы (реже в «LLM-агентах», чаще в теории/сложных workflow).

### 4) Планирование действий (не «язык агента», но близко)

- **PDDL** — классическое описание действий и целей для планировщика (символическое планирование).  
  Обзор: [Planning Domain Definition Language](https://en.wikipedia.org/wiki/Planning_Domain_Definition_Language)

### 5) То, что реально чаще всего в проде

- **OpenAPI / JSON Schema** для входов-выходов каждого агента.
- **YAML/JSON workflow** (свой небольшой DSL) или **код** (Python/TS) с явным циклом и политикой ретраев.
- **Policy как конфиг**: лимиты итераций, пороги уверенности, список «кого можно дергать повторно».

## Как это описать одной фразой в дизайне

**«Supervisor-агент с явным графом состояний: Aggregate → Validate → (Emit | Replan) с лимитом итераций и детерминированными правилами выбора следующего агента.»**

Если нужно, могу предложить минимальный **JSON-схематичный** шаблон спецификации такого агента (поля состояния, условия ветвления, формат «re-prompt») под твой стек.

## Исследование и литература (обзор)

**Multi-agent conversation и код как оркестрация.** В работе по **AutoGen** формулируется фреймворк, где несколько *conversable* агентов обмениваются сообщениями; паттерны включают последовательный чат, вложенные и иерархические диалоги — это близко к инженерной реализации «супервизор + специалисты» без единого стандартизированного ACL, но с **явной программной** спецификацией сценариев (код + естественный язык в ролях).

**Рекуррентные запросы и самокоррекция.** **Reflexion** задаёт шаблон: агент получает сигнал об ошибке/качестве, формулирует *вербальную* рефлексию в память и в следующей попытке улучшает действия. Это академически обоснованный аналог вашего «повторить запросы к другим агентам»: цикл **Act → Evaluate → Verbal reflect → Re-act**, с явным ограничением числа эпизодов.

**Обзор координации LLM-агентов.** Обзор Tran et al. (см. bib) систематизирует механизмы кооперации/конкуренции, топологии peer-to-peer и централизованные схемы — полезно при выборе между «один судья» и «равные агенты».

**ReAct и инструменты.** ReAct (Yao et al.) не про мультиагентность напрямую, но задаёт канонический **чередующийся** цикл рассуждение/действие — его часто вкладывают внутрь одного узла графа, тогда как на уровень выше остаётся ваш агрегатор.

**Смежные области и перспективы.**  
- **Формальная верификация** (TLA+, model checking) — доказать отсутствие бесконечных циклов при политике повторов.  
- **BPMN / state machines** — аудит для compliance и совместной работы с людьми.  
- **Стандартизация обмена** (FIPA ACL, JSON-RPC) — межорганизационная совместимость vs скорость итераций на JSON Schema.  
- **Безопасность:** квоты вызовов подагентов, allowlist инструментов, эскалация человеку при стагнации метрик.
